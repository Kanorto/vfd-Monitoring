# vfd-Monitoring
Это система мониторинга для дисплеев покупателей, которые основаны на VFD дисплеях. Данная программа изначально разработана для VFD DISPLAY PD-2800 от ATOL.

## Заметки и напоминания
- В приложение добавлена полноценная система заметок и напоминаний для дисплея 2x20.
- Заметки можно создавать, редактировать, включать и выключать; одновременно активно не более **10** заметок.
- Каждая заметка имеет свой интервал показа, после чего автоматически появляется на дисплее и пролистывается по 20 символов в строке, пока включена.
- Напоминания поддерживают три сценария: разовое по дате/времени, ежедневное по времени и еженедельное по выбранным дням недели.
- Сработавшие напоминания имеют приоритет над обычными метриками и заметками, показываются до скрытия или отметки о выполнении и затем попадают в историю.
- GUI разделён на две части: вкладка `Заметки` и вкладка `Напоминания`, снизу доступна история скрытых, завершённых и удалённых элементов.
- В `notes_reminders.settings` конфигурации теперь вынесены понятные лимиты и параметры поведения: `limits.max_active_notes`, `limits.max_active_reminders`, `limits.max_history_items`, `display.note_page_duration_seconds`, `display.reminder_page_duration_seconds`, `defaults.note_interval_seconds`.
- Меню трея теперь пересобирается безопасно после изменений настроек и обёрнуто защитными callback-обработчиками, чтобы после настройки хоткеев, автозапуска и других параметров трей не закрывался из-за ошибок или устаревшего состояния меню.
- Добавлены новые глобальные хоткеи:
  - `Ctrl+Alt+N` — открыть заметки;
  - `Ctrl+Alt+R` — открыть напоминания;
  - `Ctrl+Left` / `Ctrl+Right` — листать активные элементы;
  - `Ctrl+Down` — скрыть активный элемент;
  - `Ctrl+Up` — пометить активный элемент выполненным.
- Многофайловая структура Python не мешает сборке в `--onefile`: `PyInstaller` автоматически упаковывает импортируемые модули (`notes_engine.py`, `notes_gui.py`, `app_hotkeys.py`, `vfd_display.py`) внутрь одного `.exe`, поэтому приложение по-прежнему можно выпускать одним исполняемым файлом.
- Код разделён по категориям: `notes_engine.py` — заметки/напоминания и планировщик, `notes_gui.py` — GUI, `app_hotkeys.py` — горячие клавиши, `vfd_display.py` — унифицированный рендер и форматирование данных VFD.

## Автообновления
- Приложение теперь умеет проверять GitHub Releases, скачивать новый релиз и применять его при следующем запуске или во время выхода из трея.
- По умолчанию используется репозиторий `Kanorto/vfd-Monitoring`, но при необходимости `github_repo` можно переопределить в `vfd_config.json` в формате `owner/repo` или ссылкой на GitHub-репозиторий.
- Приложение подтягивает и кэширует changelog последних **10** релизов из GitHub Releases, поэтому в трее можно открыть не только последний changelog, а целую историю версий с явным разделением по релизам.
- Если автообновление несколько раз подряд не сможет скачать релиз или проверить его, приложение покажет alert со ссылкой на страницу релиза для ручного обновления.
- На стартовом экране вторая строка теперь показывает текущую версию программы.
- Автообновление теперь принимает **только** готовый `.exe` из GitHub Release и при установке всегда запускает именно этот бинарник. Поэтому клиенту не нужен установленный Python: обновление не опирается на `.py`-скрипты и не пытается запускать новую версию через `python.exe`.


## Пользовательское форматирование
- В `vfd_config.json` добавлены разделы `metric_formats` и `line_spacing`.
- `metric_formats` управляет шаблонами строк для `cpu`, `gpu`, `ram`, `disk` и `network`.
- Для CPU/GPU доступны плейсхолдеры `{usage}`, `{temp}`, `{full_prefix}`, `{short_prefix}`, `{degree}`.
- Для RAM доступен `{value}`.
- Для диска доступны `{read}` и `{write}`, для сети — `{recv}` и `{send}`.
- `line_spacing.primary` и `line_spacing.secondary` задают основной отступ между блоками на первой и второй строках.
- `line_spacing.primary_compact` и `line_spacing.secondary_compact` задают отступ, который используется, когда строка уже не помещается и приложение переходит в компактный режим.
- Быстро открыть конфиг можно из трея через пункт `Открыть конфиг`.

## Источники датчиков CPU/GPU
- Для **загрузки GPU** приложение теперь пробует несколько источников по очереди: `NVML`, затем `nvidia-smi`, затем системные Windows-счётчики `Win32_PerfFormattedData_GPUPerformanceCounters_GPUEngine`.
- Это особенно полезно для старых видеокарт NVIDIA, где `NVML` может не отдавать `utilization.gpu`, но `nvidia-smi` или системные счётчики всё ещё способны вернуть процент загрузки.
- Для **температуры CPU** приложение автоматически перебирает несколько вариантов: `LibreHardwareMonitor` WMI, `OpenHardwareMonitor` WMI, `psutil.sensors_temperatures()` и классический ACPI/WMI (`MSAcpi_ThermalZoneTemperature`).
- Для **температуры GPU** сначала используются `NVML`/`nvidia-smi`, а если драйвер или карта не отдают температуру напрямую, приложение пытается взять её из `LibreHardwareMonitor` или `OpenHardwareMonitor`.
- Если вы хотите максимум совместимости по температурам на нестандартных материнских платах и ноутбуках, удобно держать запущенным `LibreHardwareMonitor` или `OpenHardwareMonitor`: приложение само подхватит их WMI-сенсоры без дополнительной настройки.
- В `vfd_monitor.log` теперь пишется не только источник успешно выбранного датчика, но и причина отказа каждого fallback-шага. Это удобно для диагностики ситуаций вроде «`NVML` загрузку не отдаёт», «`nvidia-smi` не найден в `PATH`», «WMI namespace недоступен» или «датчики есть, но значения вне диапазона».
